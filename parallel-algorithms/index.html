Parallel Algorithms

Parallel Algorithms: Part 1.

This blog is part one of the two part series on parallel algorithms and deals with analysis of parallel algorithms. Analysing parallel algorithms are critical especially when you are trying to model how much did throwing resources at the problem actually make a difference ? The theoretical concept of doubling up on resources doesn’t really double up the speed.

An Analysis

-> Representation

The first step towards analysing algorithms is finding a way to represent them. Computation can be represented by a direct acyclic graph or a DAG wherein each node is a piece of computation work or task. Each edge is a dependency which states that a given task cannot start until all of its edges have finished execution.

A good DAG is the one that has relatively few dependencies in proportion to tasks. The figure below shows one such DAG. The nodes 2 and 3 can start executing as soon as 1 in finished and so on.

Now that we have represented the algorithm via a directed acyclic graph, let’s analyse it. Some terms:

•	Work: The work essentially means the same as for sequential algorithms except for the fact that it is divided amongst “P” processors. Work for parallel algorithms is the total amount of computation done by “P” processors. To calculate the work in a DAG, represented by W(n), just count the number of nodes. For the DAG that we have here, W(n): 8 ( assuming each node takes 1 unit time, all P’s run at the same speed and there are no edge costs for simplicity )

•	 Span: The longest path from the start to the exit in the DAG which ideally is the number of vertices on the longest path in the DAG. Therefore, if you have  to have a lower bound on the amount of time a parallel algorithm will take, its essentially the min of { Work / P,  Span }.

So, why bother about work, spans and DAGs? Turns out, the ratio of work/ span is actually the amount of parallelism available in your algorithm. This ideally means, if you have W(n) amount of work, then it would be great to use W(n)/D(n) number of processors to keep them busy on an average.

-> The key ideas to efficient parallel algorithms

Now that we have some number on how many resources to throw at the problem, let’s talk about some parallel metrics. We are of course, very interested to know how much performance gain is achieved over a sequential run of the same problem at hand. This is called speedup. The ideal speedup is of course, P times faster than the sequential time. That is, if I perform a matrix multiply on a single processor in 1 sec, using 2 processors my ideal time is 0.5 secs. This, of course isn’t true and is considered a goal, hence its an “ideal”.

With linear scaling as our goal and primary basis for design, here are some key points to keep in mind while parallelising computations and analysing a DAG:

0.	 Do not compromise work: The work done by a parallel algorithm should always match the one done by the best sequential algorithm. If you get a very parallel algorithm by increasing computation a great deal, that’s going to turn out horribly for speedup.
0.	Grow the work per processor as a function of n: As you increase the concurrency of the machine, in order to get good scaling, you might need to increase the problem size.

Good parallel algorithms are work optimal and have a low span. The best algorithms perform the task in linear time O(n). To achieve closely linear parallelism, span must be low, preferably poly-logarithm, such that the work/span ratio grows with n, close to “linearly”.

That’s all, folks ! Until next time.
